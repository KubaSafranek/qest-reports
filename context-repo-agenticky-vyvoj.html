Context repo pro agentický vývoj

Celkový obraz: jak je repo uspořádané, jak jím protéká práce, a co přesně je v každém .md souboru. Společná pravidla: Markdown + YAML frontmatter, lidský souhrn nahoře, agentský detail pod ním, stabilní ID a křížové odkazy. Příklady na datech Smart Redirection.

1 — Přehled

Celková struktura repozitáře

Vrstva context/ drží veškerý kontext pro agenta vedle kódu. Sedm složek odpovídá sedmi otázkám: proč, co, jak, proč zrovna takto, co dělat, jak to stavět a co je kolem.

<service>/                       # repo služby (např. smart-redirection-node)
├─ AGENTS.md                      # vstup pro AGENTA: navigace + zlatá pravidla
├─ README.md                      # vstup pro ČLOVĚKA
│
├─ context/                       # ── AI KONTEXTOVÁ VRSTVA (zdroj pravdy) ──
│  ├─ product/                     # PROČ — vision.md, glossary.md, ideas/
│  ├─ requirements/                # CO — SR-REQ-*.md (1 soubor = 1 požadavek + AC)
│  ├─ architecture/                # JAK — overview, domain-model, data-model, integrations
│  ├─ decisions/                   # PROČ TAK — ADR-*.md (rationale trail)
│  ├─ stories/                     # PRÁCE — SR-US-*.md → generuje ADO work item
│  ├─ conventions/                 # STAVĚT — coding-standards, definition-of-done, guardrails
│  └─ external/                    # SVĚT — výtahy o okolních službách
│
├─ schema/                        # JSON schémata pro validaci frontmatteru
├─ src/  (core/ infrastructure/ interface/)   # samotný kód
└─ tests/

2 — Proces

Jak repem protéká práce

Smyčka spec-driven vývoje. Zdrojem pravdy je strukturované zadání v GITu; agent ho atomizuje, validuje, implementuje a vývojář schvaluje. Azure DevOps je status mirror, ne místo, kde se píše zadání.

  1. Zadání → atomizace.
    Agent převede prózu (PDF na Drive, Notion Návrh) na requirements/*.md + decisions/*.md s ID a traceability na zdroj (source:).
  2. Validace struktury.
    Pre-commit hook + CI gate ověří frontmatter proti schématu a referenční integritu (covers/depends_on/relates_to ukazují na existující ID). Nevalidní kontext neprojde merge.
  3. context/ = zdroj pravdy.
    GIT drží CO/JAK/PROČ. Je to baseline, proti kterému se měří změny i progress.
  4. Generování práce do ADO.
    Ze story v GITu vznikne ADO work item — tenký ukazatel: lidský souhrn + odkaz na soubor v GITu. GIT je master, stav žije v ADO.
  5. Agent implementuje.
    Kódovací agent čte navázaný requirement + jeho ADR + AGENTS.md, napíše kód a testy podle acceptance_criteria, otevře Pull Request.
  6. Vývojář reviduje.
    Jediný povinný lidský krok. Spec i kód jsou v jednom diffu → reviewer vidí úplný kontext. Schválí → merge.
  7. Zpětná vazba.
    Stav z ADO + git diff requirements/ → progress proti pokrytí požadavků a change-log / scope audit zadarmo.

Co je v každém souboru

ENTRY

Vstupní body

/
AGENTS.md
První, co agent přečte. Navigační mapa kontextu + zlatá pravidla. Krátké, odkazuje, neduplikuje. Načítá se automaticky.
Příklad
# Smart Redirection — agent context

Mikroslužba, která automaticky přesměrovává zásilky na vhodnější PUDO.
Záměr a hranice → context/product/vision.md

## Mapa kontextu
- Co stavíme → context/requirements/   (REQ s akceptačními kritérii)
- Jak → context/architecture/         (model, data, eventy)
- Proč tak → context/decisions/          (ADR — nevracej zamítnuté)
- Práce → context/stories/              (mapuje na ADO)
- Pravidla → context/conventions/        (standardy, DoD, guardrails)

## Zlatá pravidla
1. Než začneš úkol: přečti navázaný requirement + jeho `relates_to` ADR.
2. Nejednoznačné zadání → zeptej se, neimprovizuj (viz agent-guardrails.md).
3. Po dokončení aktualizuj `status:` a `implemented_in:` v requirementu.
4. Kód musí splnit `acceptance_criteria` — ne víc, ne míň.
README.md
Vstup pro člověka. Co služba je, jak ji lokálně spustit, kde je kontext. Krátké — detail je v context/.
Čistě lidský dokument, agent ho nepotřebuje. Drž ho stručný a odkaž na AGENTS.md.

# Vize — Smart Redirection

Souhrn: Snížit náklady na OOH doručení a zvýšit vytíženost DPD boxů
automatickým přesměrováním zásilek na vhodnější výdejní místo v okolí.

## Cíle
- Přesměrovat zásilku na levnější/bližší PUDO ještě před výjezdem kurýra.
- Preferovat DPD boxy před konkurenční sítí (úspora poplatku partnerovi).

## Hranice (co služba NEdělá)
- Negarantuje rezervaci — pracuje jen s aktuálně dostupnou kapacitou.
- Neřeší fyzický svoz ani trasování kurýra (to je doména kurýrní sítě).
- Neoptimalizuje spokojenost zákazníka — ta je jen výstupní metrika.

# ADR-002: XL kapacita pouze AlzaBox + DPD Box

Rozhodnutí: XL schránky existují pouze v sítích DPD Box a AlzaBox.
Ostatní boxy (Z-BOX, GLS, One Box, OX Point) XL kapacitu nemají.

Důvod: Odpovídá reálnému stavu boxů v praxi.

## Důsledky
- Generátor zásilek filtruje: XL cíl jen z lokací s XL kapacitou + všechny VM.
- XL přesměrování má vlastní větev (Krok 0) mimo hlavní cyklus.
- Brand bez XL kapacity NIKDY není cílem XL zásilky → fail-fast assertion.

Příklad
# Coding standards

- Hexagonální architektura: `core/` (doména) · `infrastructure/` · `interface/`.
- Use-case pattern — use-case NESMÍ volat jiný use-case; sdílené → Services.
- Doménová logika v entitách, ne v service/repository. Zákaz „utils" modulů.
- Style: 4 mezery, single quotes, bez středníků, žádné implicitní `any`.
conventions/definition-of-done.md
Kdy je práce hotová. Checklist, který musí agent i reviewer projít před uzavřením story.
Příklad
# Definition of Done

- [ ] Všechna `acceptance_criteria` navázaného requirementu mají test a procházejí.
- [ ] `status:` requirementu posunut na `implemented`, `implemented_in:` vyplněn.
- [ ] Žádná změna chování bez navázaného requirementu nebo ADR.
- [ ] Lint + typecheck čisté; PR má odkaz na story i requirement.
conventions/agent-guardrails.md
Mantinely pro agenta. Co smí měnit sám, co ne, a kdy se MÁ zeptat místo improvizace. Klíčové pro to, aby agent neudělal škodu mimo scope.
Příklad
# Agent guardrails

## Smíš sám
- Implementovat kód dle requirementu a jeho acceptance_criteria.
- Přidat/upravit testy. Aktualizovat `status:` a `implemented_in:`.

## Nesmíš bez schválení
- Měnit `acceptance_criteria` nebo znění requirementu.
- Měnit integrační kontrakty (eventy, API) — to je ADR + souhlas.
- Mazat nebo přepisovat ADR.

## Zeptej se, když
- Zadání je nejednoznačné nebo si protiřečí s ADR → STOP, zeptej se.
- Requirement nemá acceptance_criteria → nedá se ověřit hotovo, vyžádej je.

4 — Challenge

Kde to naráží na dnešní standardní proces

Dnešní proces je sekvenční (BA → Architekt → PM → Dev → Tester), dokumenty jsou próza, ADO je centrum a odhady stojí na člověkohodinách. Agent-first model to na několika místech láme — pojmenovat to teď, ne až v pilotu.

5 — Kde je čára

GIT vs Notion — co kam patří

Cílem není „veškeré produktové dokumenty v repu". Hodnota je koncentrovaná do kontraktní vrstvy; u živé vrstvy klesá do záporu. Rámec: rozdělujeme dokumentaci na kontraktní (repo) a živou (Notion).

Heuristika: Do repa patří dokument tehdy, když správnost agenta závisí na tom, že je verzovaný atomicky s kódem. V Notionu zůstává, když je primární use-case lidská kolaborace nad rychle se měnícím obsahem.
do repa — kontraktní vrstva

GIT

  • requirements + akceptační kritéria
  • ADR (rozhodnutí), domain model
  • glossary, integrační kontrakty (eventy, API)
  • conventions, agent-guardrails, architektura
do notionu — živá vrstva

Notion

  • budget, RAID / rizika, roadmapa
  • stakeholder mapa, status reporty
  • meeting zápisy
  • klientské materiály
Co repo-vrstva koupí (o agentovi, ne teorie)Co by se ztratilo nacpáním živé vrstvy do gitu
Stejný kontext v jednom diffu — bez MCP round-tripů, levné na tokeny.Real-time kolaborace a přístup ne-vývojářů.
Atomicita spec↔kód — spec nemůže tiše driftovat.Bohaté views / databáze / rollupy.
Review ve stejné bráně (PR + CODEOWNERS).Nízká friction častých úprav.
Pravidla jako gate (schema, „merge neprojde bez testu").Žádný šum z volatility v commit historii.
Audit „proč" (ADR + git blame); větvení specu vedle kódu.Granularita přístupu; stale dokument v repu je nebezpečnější (agent mu věří).
Pro meeting: neprezentovat jako „migrujeme docs do repa", ale jako „rozdělujeme dokumentaci na kontraktní a živou" — obhajitelné a nezabíjí to, co na Notionu funguje.

6 — Stav

Vizualizace stavu zůstává na boardu

PM/BA chtějí vidět, v jakém stavu co je. To git neřeší a řešit nemá — je to rychlá, volatilní vrstva.

  • Git = kontrakt (co a proč stavíme). Board = stav a tok (kde to je).
  • Story v gitu nese odkaz ado: → board v Azure DevOps ukazuje stav, sprint, flow, burndown.
  • Notion drží filtrované views nad živou vrstvou (budget, RAID, roadmapa) — co board v ADO neumí.
  • Pravidlo: stav teče z boardu, kontrakt z gitu; nic stavového se do gitu nezrcadlí.

  • Testy z AC, nezávisle: acceptance testy píše jiná entita než implementační agent (Tester nebo oddělený „QA agent" kurátorovaný Testerem) — jinak agent „známkuje vlastní úkol".
  • Role Testera: hlídá testovatelnost AC, vlastní acceptance suite, dělá exploratory na to, co AC nepokryla, drží test-gate a finální sign-off.
  • Definition of Done váže merge na pokrytí AC; automatický suite je bezpečnostní síť rychlé agentí iterace.

9 — Migrace

Jak začít — fázovaně, ne big-bang

  1. Fáze 0 — základ.
    Vybrat 1 službu + 1 squad (pilot: Smart Redirection, kde už je AGENTS.md). Postavit šablony, schéma, validační gate.
  2. Fáze 1 — shadow run.
    BA atomizuje jeden epic do requirements+AC, Architekt doplní architecture + ADR. Běží paralelně se současným procesem → nízké riziko, srovnání.
  3. Fáze 2 — první stories agentem.
    Tester připraví acceptance testy, agent implementuje, Dev reviduje. Měřit metriky.
  4. Fáze 3 — retro & go/no-go.
    Doladit šablony, guardrails, model testů; rozhodnout pokračování.
  5. Fáze 4 — rozšíření.
    Na celou službu, pak další squad. Legacy kontext doplňovat oportunisticky (jen při doteku), ne najednou. ADO zůstává — žádný hard cut-over.

10 — Nacenění

Jak fakturovat a naceňovat

Účtovat „kódovací MD" se rozpadá — když agent hodiny zkrátí, propadne revenue při stejné hodnotě. Rozdělit cenu na tři složky:

SložkaCo to jeJak účtovat
Lidská expertízaBA, Architekt, Dev-reviewer, Tester, PM — tady reálná práce zůstávákapacita / MD (klasicky)
Delivery jednotkadodaný requirement / storyfixní cena za story dle složitosti (S/M/L), kalibrovaná z pilotu — nahrazuje „kódovací MD"
Platforma / agentnáklad na agenta (compute)pass-through nebo paušál, jako samostatná položka
  • Pro klienta: platí za výsledky + expertízu + platformu, ne za hodiny ťukání.
  • Odhady v počtu stories, ne v MD. Pilot změří throughput a cenu agenta na story → z toho ceník složitostních tříd.
  • Otevřené: marže na platformě vs. pass-through; jak ošetřit rework ve fixní ceně za story.

12 — Most

Kde se řeší rizika, meetingy a otázky

  • Rizika (RAID), meeting zápisy, otevřené otázky, status → Notion (živá vrstva, board views).
  • Git dostává jen destilát: když diskuse vyprodukuje rozhodnutí ovlivňující chování → vznikne ADR nebo úprava requirementu v gitu; zápis zůstává v Notionu.
  • Governance: risk review a steering běží nad Notionem; architektonická a kontraktní rozhodnutí povinně končí jako ADR/requirement v gitu.
Pravidlo mostu: Notion je místo diskuse a živého stavu; git je místo destilovaného kontraktu. Diskuse v Notionu → rozhodnutí v gitu (ADR/requirement) → odkaz zpět.

13 — Governance

Řízení a brány kvality

  • Change control: změna requirementu = PR + schválení; architektonická změna = povinné ADR; scope change je vidět jako git diff.
  • 3 quality gates: validační (struktura) → review (člověk) → test-gate (pokrytí AC). Definition of Done je váže.

  • Kadence: spec grooming (BA+Architekt), SLA na review, retro na samotný proces, přehled metrik + steering pilotu.
