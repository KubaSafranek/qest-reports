Datový model · topologie repozitářů · co kam patří

Jak to postavit od dat

Tři úrovně domény (Projekt × Produkt × Service) napříč mnoha repozitáři. Registr identit a vztahů už existuje v Notionu (Products DB, Service Catalog, Projects DB). Na něm stavíme: kontrakt do GIT, stav na Board, živá vrstva a registr v Notion.

1 — Doména

Tři úrovně: Projekt × Produkt × Service

Tohle je páteř celého modelu — bez něj nejde rozhodnout, co kam patří. Úrovně se kříží: jeden projekt zasahuje do více produktů, jeden produkt sdílí více services.

aktivita · cross-product

Projekt

Iniciativa zlepšující proces, většinou napříč produkty (může být i jedno-produktová). Má cíl, stakeholdery, časový rámec.

Smart Redirection — zasahuje DPD Box, Boxy 3 stran, My Pickup; prolíná různé uživatele.
aplikace · jasní uživatelé

Produkt

Ucelená aplikace s jasnými uživateli a Product Ownerem. Skládá se ze sady services.

Locker-service (DPD-BOX) — PO, Status „In Production", repos lockers-node, universe-react.
komponenta · sdílitelná

Service

Nasaditelná komponenta, kterou může sdílet více produktů. Má tech leada, severity, prostředí.

LockersReservationApiProd — „API for receiving reservations from MID", ECS AWS, Severity A.

4 — Topologie

Struktura repozitářů — mnoho repů + jeden hub

Klíčové napětí: projekt je cross-repo (zasahuje víc produktů/services), ale chcete jeden repozitář pro projektové záležitosti. Řešení = dvě vrstvy, ne konflikt: projektový kontext do hubu (cross-repo), service kontext zůstává u kódu (lokální).

Hub — jeden repo pro projekty

project-hub/ # 1 repo, domov PM/BA ├─ projects/ │ └─ smart-redirection/ │ ├─ vision.md │ ├─ requirements/ # cross-product REQ+AC │ ├─ decisions/ # projektové ADR │ ├─ stories.md # index → Board + repo │ └─ affects.yaml # dotčené produkty/services ├─ registry/ # zrcadlo IDs z Notionu │ ├─ products.yaml │ └─ services.yaml # ID → repo URL └─ glossary.md

Service repo — kontext u kódu

lockers-node/ # existující repo služby ├─ AGENTS.md ├─ context/ │ ├─ architecture/ # overview, data-model │ │ └─ integrations.md # API/eventy (kontrakt) │ ├─ decisions/ # service ADR │ ├─ requirements/ # jedno-service REQ │ └─ conventions/ # standards, guardrails ├─ core/ infrastructure/ interface/ └─ tests/
Most přes ID: requirement v hubu nese targets: [SERVICE-ID] + odkaz na repo z registry/services.yaml; kód v service repu nese REQ-ID. Pravidlo umístění: cross-product požadavek → hub; jedno-service požadavek → service repo. Mnoho repů nekonsolidujeme — konsolidujeme jen projektový kontext a registr.

5 — Proces

Flow: BA → PM + Tech Lead → QA

Každá role má vstupní nástroj a jasný výstup. Stav je vždy na boardu, kontrakt vždy v gitu.

  1. Business analytik
    Notion → GIT
    Ověří Projekt a dotčené produkty/services (Notion), pak napíše requirements + AC do hubu (git).
  2. PM
    GIT → Board
    Z requirementů založí epics/stories na boardu, plánuje, hlídá stav a scope (git-diff).
  3. Tech Lead
    GIT (service repo)
    Architektura + ADR, rozpad na service stories přes dotčené repo, nastaví guardrails. Pak agent + Dev review.
  4. QA
    GIT + Board + Notion
    Z AC acceptance testy (nezávisle), drží test-gate a sign-off; test plány do Notionu.
Pořadí předávek: BA (co a proč) → PM (plán a stav) + Tech Lead (jak) → agent (implementace) → Dev (review) → QA (kvalita a sign-off). Lidské rozhodovací body: AC, architektura, PR review, test-gate.

6 — Vztah

Hub ↔ Service repo ↔ Board — kdo co edituje

Hub je per-projekt (dočasný, cross-repo), service repo je per-služba (trvalý, sdílený mnoha projekty). Spojují se přes ID, obsah se nekopíruje. „1 projekt → N služeb" a zároveň „1 služba ← N projektů".

targets — hub říká „týká se služby X" covers — service říká „plním REQ Y" story ↔ ticket — stav na boardu
Životní cyklus: hub-složka projektu se po dokončení zavře, ale co projekt přidal do service/context/ (kontrakt, ADR) zůstává a obohacuje službu pro další projekty. Proto je service repo dlouhověký a sdílený, hub dočasný.
